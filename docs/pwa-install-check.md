# PWA 홈 화면 설치 확인 절차

배포 후(또는 로컬 프로덕션 빌드 후) 실기기에서 한 번씩 확인한다. 자동화된 테스트가 없는
영역이라(브라우저/OS 동작) 이 문서가 사실상의 테스트 목록이다.

## 테스트 환경 — localhost와 실기기는 다르다

서비스워커/PWA 설치 판정은 브라우저가 "보안 컨텍스트"(HTTPS 또는 `localhost`)를 요구한다.

- **같은 컴퓨터의 데스크톱 브라우저로 `http://localhost:3000`** — `localhost`는 예외로
  보안 컨텍스트 취급되므로 서비스워커 등록·manifest·오프라인 캐싱 devtools 확인은 배포 없이
  로컬에서 전부 검증 가능하다.
- **폰으로 `http://<랩탑의 LAN IP>:3000`(같은 와이파이)** — `localhost`도 HTTPS도 아니라서
  보안 컨텍스트 조건을 못 만족한다. 서비스워커 등록 자체가 안 되거나 설치 배너가 안 뜰 수
  있다 — 이 방식으로는 아래 iOS/Android 실기기 절차를 신뢰할 수 없다.
- **실기기에서 "홈 화면에 추가"를 제대로 확인하려면 실제 HTTPS URL이 필요하다** — Vercel
  Preview/Production 배포(`docs/deploy.md`, Preview 환경에도 `GATE_PASSWORD` 설정 필요) 또는
  ngrok 같은 HTTPS 터널을 로컬 서버에 연결한다.

## iOS Safari

1. 배포된 URL을 Safari로 연다 → 게이트 로그인.
2. 공유 버튼(사각형 + 위쪽 화살표) → "홈 화면에 추가".
3. 미리보기에 표시되는 이름이 "다아람"인지, 아이콘이 흐릿한 스크린샷이 아니라
   보라색 배경 + 흰 "다" 글자로 된 아이콘인지 확인한다. (아이콘이 스크린샷으로 나오면
   `/apple-icon`이 빌드에 반영 안 됐거나 캐시 문제 — Safari 설정 → 고급 → 웹 사이트 데이터에서
   해당 사이트 삭제 후 재시도)
4. 홈 화면 아이콘 탭 → 주소창/탭 바 없이 전체화면(standalone)으로 뜨는지 확인.
5. 앱을 한 번 정상적으로 열어 몇 개 화면(레시피 목록 등)을 돌아본 뒤 종료.
6. 설정 → 기내 모드 켜기.
7. 홈 화면 아이콘으로 다시 실행 → 방금 방문했던 화면이 흰 화면/에러 없이 뜨는지 확인.
   (뜨지 않으면 서비스워커가 `/_next/static/*` 청크를 못 캐싱한 것 — 아래 "문제 해결" 참고)
8. 기내 모드 해제.

## Android Chrome

1. 배포된 URL을 Chrome으로 연다 → 게이트 로그인.
2. 주소창 오른쪽 점 세 개 메뉴 → "앱 설치" (또는 하단에 자동으로 뜨는 설치 배너의 "설치" 버튼).
3. 설치 다이얼로그에 이름/아이콘이 올바르게 나오는지 확인.
4. 설치 후 앱 서랍/홈 화면 아이콘으로 실행 → 브라우저 UI 없이 뜨는지 확인.
5. iOS와 동일하게 몇 화면 방문 후 기기를 비행기 모드로 전환, 앱을 다시 열어 오프라인에서도
   뜨는지 확인.

## 문제 해결 (devtools)

- **manifest 파싱 에러**: Chrome devtools → Application 탭 → Manifest. 에러/경고가 있으면
  `public/manifest.json`을 확인한다.
- **서비스워커 등록 상태**: Application 탭 → Service Workers. "activated and is running"이
  아니면 새로고침(하드 리프레시) 후 다시 확인.
- **캐시된 파일 확인**: Application 탭 → Cache Storage → `app-shell-vN`. `/`, `/manifest.json`,
  `/icons/icon.svg`와 방문했던 페이지의 `/_next/static/...` 청크들이 들어있어야 한다.
- **구버전 캐시가 안 지워짐**: `public/sw.js`의 `CACHE_VERSION`을 올려서 배포했는지 확인 —
  버전을 안 올리면 `activate` 단계의 구버전 캐시 삭제 로직이 트리거되지 않는다.

## 알려진 한계

- 한 번도 방문한 적 없는 라우트는 오프라인에서 열리지 않는다 (그 라우트의 JS 청크가
  아직 캐시에 없기 때문). "앱 셸만 캐시"라는 설계상 의도된 동작이며, 데이터(IndexedDB)와
  달리 방문 이력이 없는 화면까지 미리 채워 넣지는 않는다.
- 로그인 전(게이트 통과 전) 상태에서 서비스워커가 최초 설치되면 `/`이 게이트 리다이렉트
  응답으로 캐시될 가능성이 있다 — 알려진 상태지만 이번 작업 범위에서 고치지 않았다
  (docs/decisions.md 참고).
