# Plan and Review

이 프로젝트는 LLM을 쓰지 않으므로 LLM 관련 체크리스트는 없다. 대신 IndexedDB 마이그레이션·
백업 호환성·오프라인 동작 체크리스트가 있다 (서비스 구상서 §7 프로젝트 고유 위협 반영).

## 목표
구현 전에 검증 가능한 계획을 만든다. 이 스킬이 호출된 턴에서는 코드를 구현하지 않는다.

## 절차

### 1. 문제 재진술
요청을 한 문장으로 재진술하고, 모호한 부분이 있으면 가정하지 말고 질문한다.

### 2. 출력 계약 먼저
최종 결과의 형태를 먼저 정의한다: TS 함수 시그니처(반환 타입), zod 스키마(경계인 경우),
예시 값, 실패 케이스(어떤 입력이 어떤 오류로 드러나는지).

### 3. 입력 모델
필요한 최소 입력을 TS 타입으로 정의한다. 원시 타입이 도메인 값이면 branded type 후보로 표시한다.

### 4. 제약과 금지 상태 (Deny First)
허용되면 안 되는 입력·상태를 나열하고, 각각을 "타입으로 막을 것(판별 유니온·branded)"과
"zod 런타임 검증으로 막을 것"으로 분류한다. 타입으로 막을 수 있으면 타입이 우선이다.

### 5. 변경 지점
나중에 바뀔 가능성이 높은 규칙(카테고리 정책, 원가 계산 규칙 등)을 식별한다. 변경 지점만
lib/policy/의 인터페이스 뒤로 분리한다. 변경 근거가 없는 것에 인터페이스를 만들지 않는다 (과잉설계 금지).

### 6. 책임 분리
요구사항 → 책임 → 모듈(lib/domain / lib/policy / app+store / lib/infra 중 어디) 매핑 표를 만든다.
IndexedDB 접근이 포함되면 반드시 lib/infra/db.ts 경유로 표시한다.

### 7. 자기검토 (구현 전 필수)
다음 질문에 답한다:
- 출력을 구현보다 먼저 정의했는가?
- 스펙에 없는 규칙을 가정으로 심었는가? 있다면 [가정 — 확인 필요]로 표시했는가?
- 구현이 하나뿐인데 인터페이스를 만든 곳이 있는가? (있으면 강등)
- 각 실패 케이스에 대응하는 테스트가 목록에 있는가?
- app/ 또는 store/에 계산 로직이 들어갔는가? (있으면 lib/policy 또는 lib/domain으로 이동)
- IndexedDB 스키마 변경이 포함되는가? [F6] 포함된다면:
  - Dexie upgrade 함수로 마이그레이션 경로를 작성했는가?
  - 기존 데이터가 있는 상태에서 업그레이드하는 테스트가 계획에 있는가?
- 백업 export/import 형식에 영향을 주는가? [F5] 영향을 준다면:
  - 이전 버전 백업 파일을 import했을 때의 동작이 계획에 명시돼 있는가?
  - import 시 zod 스키마 검증을 거치는 경로인가?
- 오프라인(서비스워커 캐시) 동작에 영향을 주는가? [F7] 영향을 준다면 캐시 무효화 전략을 명시했는가?

### 8. 테스트 목록
happy path / 빈 입력 / 잘못된 입력 / 경계값 / 각 정책 케이스를 나열한다.
각 테스트에 "검증하는 요구사항"을 한 줄로 붙인다. IndexedDB 의존 테스트는 fake-indexeddb를 명시한다.

### 9. 종료
"이 계획을 검토해 주세요. 승인되면 테스트부터 작성합니다(red)."로 끝낸다.
